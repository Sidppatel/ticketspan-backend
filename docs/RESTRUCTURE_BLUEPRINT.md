# Svyne Backend Restructure Blueprint

Single source of truth for the full restructure. No code is written until this is approved.

Decisions locked in:

- Delivery: **plan only first** (this document).
- Work order once approved: **(1) DB schema + unified users → (2) proto contracts → (3) folder/repo layout**.
- Database history: **reset clean** — drop all existing migrations, create one fresh `InitialCreate` against the new unified schema. Existing dev data is discarded.
- Transport: **gRPC-Web over HTTPS** replaces every REST controller. File upload stays multipart HTTPS (RULES Communication §1 File Upload Exception).

## 1. Phase 1 — Database schema + unified users

### 1.1 `tenants` (renamed from `organizations`)

- PK `tenants_id UUID`.
- `slug` (subdomain), `name`, `stripe_connected_account_id`, status fields, `created_at`, `updated_at`.
- Every tenant-scoped table's `organization_id` → `tenants_id` (RULES Rule 10/11).

### 1.2 `users` (merge of `users` + `business_users`)

Single table. Columns (snake_case per Rule 10):

|--------|------|-------|

| `first_name`, `last_name` | text | |
| `password_hash` | text NULL | NULL for Google-only |

| `google_subject` | text NULL **UNIQUE** | OAuth linkage, security-critical unique |
| `role` | SMALLINT | 0 Attendee, 1 Main Admin, 2 Staff, 3 Sub-Tenant, 99 Developer |
| `phone` | text NULL | |
| `avatar_url` / `image_id` | — | keep `image_id` FK (matches existing Image table) |

| `email_verified` | bool default false | + `email_verified_at` |
| `last_login_at`, `last_request_at` | timestamptz NULL | |

| `created_at`, `updated_at` | timestamptz | |

Constraints:
- `UNIQUE (tenants_id, email, role)` — same email across tenants + dev (NULL tenant) coexist.
- `CHECK (role = 99) = (tenants_id IS NULL)` — developers global, everyone else tenant-bound.
- `UNIQUE (google_subject)` partial where not null.

### 1.2.1 Password hashing (salt + pepper)

- Algorithm: **Argon2id** (or keep current bcrypt if already wired — TBD by existing AuthService lib).
- **Per-hash random salt** is generated by the algorithm itself → two users with the same plaintext password produce **different** `password_hash` values. No manual salt management needed.
- **Pepper**: a server-side secret applied to the password before hashing (HMAC-style), loaded from config/env, **never hardcoded** (Rule 12) and **never stored in the DB**. Defends against offline cracking if the DB leaks.
- **Versioned peppers (rotation-safe)**: switching the pepper would otherwise break every stored hash (pepper is baked in, not stored). To allow rotation:
  - `users` gains `pepper_version SMALLINT NOT NULL DEFAULT 1`.
  - Config holds a map `{ 1: PASSWORD_PEPPER_V1, 2: PASSWORD_PEPPER_V2, ... }` (env vars `PASSWORD_PEPPER_V<n>`), plus `PASSWORD_PEPPER_CURRENT = <latest n>`.
  - **Hash/write**: always use the current version; persist its number in `pepper_version` via `sp_set_user_password` / `sp_create_user`.
  - **Verify**: look up the pepper by the row's `pepper_version`, then algorithm-verify.
  - **Rehash-on-login**: on successful login, if `pepper_version < current`, recompute hash with current pepper and update the row (lazy migration). Argon2 param upgrades ride the same path.
  - Rotating = add `PASSWORD_PEPPER_V<n+1>`, bump `PASSWORD_PEPPER_CURRENT`; old hashes keep verifying, users migrate on next login. Old pepper retired once no rows reference its version.
- Hashing happens in C# AuthService before calling `sp_set_user_password` / `sp_create_user`; SPs only ever receive/store the final hash + `pepper_version`, never plaintext.

### 1.4 RLS with developer bypass (RULES Rule 11)

- `app.current_user_id()` → reads `current_setting('app.current_user_id')::uuid`.
- Helper `app.is_developer()` → `(SELECT role FROM users WHERE users_id = app.current_user_id()) = 99`.
- Per tenant-scoped table policy:

```
USING ( app.is_developer() OR tenants_id = current_setting('app.current_tenant')::uuid )
```

- One `.sql` file per policy under `database-scripts/policies/` (Rule 14).

### 1.5 Stored procedure rewrite (the heavy part — 166 SPs)

Mechanical rules applied across all SPs:
- `p_business_user_id` / `p_user_id` → **`p_users_id`** (unified).
- Any read/write of `business_users` → `users` with `role` filter.
- `organization_id` → `tenants_id`.
- Drop now-redundant SPs (merge pairs): e.g. `sp_get_business_user_by_id` + `sp_get_user_by_id` → `sp_get_user_by_id`; `sp_create_business_user` + `sp_signup_user` → `sp_create_user`; session/password-reset/avatar `*_business_user_*` pairs collapse to single `*_user_*`.
- `sp_create_organization` → `sp_create_tenant`: inserts `tenants` row **+** first admin `users` row (`role = 1`, `tenants_id` = new id) **+** issues magic-link/setup token.

Estimated SP set after merge: ~120 (≈46 business/user duplicates collapsed). A full old→new SP mapping table will be produced as `docs/SP_MIGRATION_MAP.md` during build.

### 1.6 Views

- Rename every `v_*` → `vw_*` (Rule 7).
- Merge `vw_business_users` + `vw_user_profile` → single `vw_users`.
- `vw_organizations` → `vw_tenants`.
- One file per view under `database-scripts/views/`.

### 1.7 Clean migration reset

- Delete all files in `Migrations/`.
- New `InitialCreate` builds unified schema (structure only — no seed, Rule 13).
- SQL artifacts (views/SPs/policies/triggers) installed via the existing `InstallSqlArtifacts` migration pattern reading from `database-scripts/` (Rule 15: migrations run exclusively via .NET).
- `init-data/` holds lookup/`app_settings` seed with `ON CONFLICT DO NOTHING`.

### 2.2 Message rules

- Every message field maps 1:1 from existing DTOs in `contracts/DTOs/`.
- Backend maps `vw_*` raw-SQL results → proto messages directly (RULES Communication §Integration: gRPC does not override Rules 7/8/9).
- `tenants_id`, `role` carried in JWT, not in request bodies.
- Zero comments in `.proto` (RULES Rule 1).

### 2.3 Server wiring

- `Grpc.AspNetCore` + `Grpc.AspNetCore.Web` packages.
- Replace MVC controller registration with gRPC service registration in `Program.cs`.
- `#if DEBUG` JSON transcoding for local dev only (RULES Communication §Local Dev Exemption).
- Tenant middleware: read `tenants_id`/`users_id` from JWT → `set_config('app.current_tenant',…)` + `set_config('app.current_user_id',…)` per request. If `tenants_id` null and `role != 99` → reject.

├── src/
│   └── Api/                      # the .csproj — SQL-FREE (Rule 6)
│       ├── Grpc/                 # gRPC service implementations
│       ├── Data/                 # DbContext, raw-SQL repos, entities (read views)
│       ├── Services/             # business logic
│       ├── Middleware/           # tenant resolution, RLS context, auth
│       ├── Migrations/           # EF migrations (structure only)
│       └── Helpers/ Extensions/ Validators/ Workers/
├── protos/                       # single source of truth (.proto)
├── database-scripts/
│   ├── views/                    # one vw_*.sql each (Rule 14)

│   ├── policies/                 # one RLS policy each
│   ├── triggers/                 # one trigger each
│   ├── init-data/                # lookup/app_settings seed, ON CONFLICT
│   └── seed-data/                # sample data — never run by migrations (Rule 13)

## 5. Resolved decisions

1. **JWT**: keep current signing library. Passwords: Argon2id/bcrypt per-hash random salt (auto-unique) **+** versioned server pepper (`PASSWORD_PEPPER_V<n>` env, `pepper_version` column, rehash-on-login rotation) — §1.2.1.
2. **Google OAuth**: confirmed — frontend gets id-token → backend validates `sub`+`email`, links/creates per rules doc §4.
3. **`contracts/` DTO project**: **retire**. Proto messages become the wire contract; backend maps `vw_*` raw-SQL rows → proto messages directly. No separate DTO project to maintain (avoids a redundant layer, RULES Rule 4 simple architecture). Any internal-only shapes (e.g. SP param records) live beside the service that uses them.
4. **Frontend**: out of scope. Backend only this effort. Frontend regenerates stubs from `protos/` later.
